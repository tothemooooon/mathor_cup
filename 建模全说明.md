# MathorCup A题建模全说明（主线版）

> 文档定位：本文件作为 A 题统一建模说明主文档，以“每一问优化后效果最好的模型”为主线，覆盖建模思路、实验设置与结果、代码实现原理、已知边界与后续优化方向。

## 1. 项目目标与评审导向

### 1.1 目标定义
- 总目标：在竞赛评分体系下最大化综合得分，而非仅追求单次最优值。
- 评估优先级：可行性 > 结果质量 > 解释性 > 工程可复现性。

### 1.2 统一评分代理
- 可行率（约束满足情况）
- 目标值（travel + 时间窗惩罚 + 车辆惩罚）
- 稳定性（多 seed 波动）
- 可解释性（建模口径、参数含义、决策过程）

## 2. 环境与复现规范

### 2.1 运行环境
- Python: 3.10
- 包管理: uv
- 环境隔离: 项目内 `.venv`
- QUBO/退火: Kaiwu SDK

### 2.2 统一入口
- 统一运行脚本：`run_baseline.py`
- 标准命令格式：

```bash
uv run python run_baseline.py --profile <profile_name> --question Q1|Q2|Q3|Q4
```

### 2.3 结果契约
- 所有结果落盘到 `experiments/results/*.json`
- 统一包含：`parameters`、`diagnostics`、`metrics`

## 3. 数据说明与质量评估

### 3.1 数据结构
- 节点属性表：节点、时间窗上下界、服务时长、需求、车辆容量
- 旅行时间矩阵：0~50 节点间时间成本

### 3.2 数据质量检查结论
- 维度完整，矩阵非负、对角为 0、与节点索引一致

## 4. 统一建模框架

### 4.1 基础变量
- 位置编码：`x[i,t]`，表示客户 `i` 位于路径位置 `t`

### 4.2 统一目标
- 基础旅行成本 + 约束惩罚（位置唯一/客户唯一）+ 业务惩罚（时间窗、车辆数）

### 4.3 四问关系
- Q1: 单车 TSP QUBO
- Q2: Q1 + 时间窗惩罚入模
- Q3: Q2 思路扩展到 50 客户（分解混合求解）
- Q4: 多车容量约束 + 车内 QUBO + 跨车协同优化

## 5. Q1 主线模型（最优版）

### 5.1 问题定义
- 15 客户单车路径最短（无时间窗、无容量约束）

### 5.2 主线模型
- 模型名称：`QUBO-TSP + Kaiwu SA (adaptive lambda + profile ensemble + anchor decomposition)`
- 核心思想：
  - 用位置编码构造 TSP QUBO
  - 自适应惩罚系数 `lambda`
  - 多退火档案（P1/P2/P3）
  - 锚点分解（固定首客户的重启策略）

### 5.3 实验设置
- 客户数：15
- `seed_count=8`
- `use_adaptive_lambda=True`
- `adaptive_rounds=5`
- 对照基线：早期单一 SA 版本

### 5.4 结果报告
- 当前最优：`travel=30, objective=30, feasible=True`
- 结果文件：`experiments/results/q1_20260417_205616.json`
- 与早期基线（`travel=47`）相比：下降约 36.17%

### 5.5 代码实现原理
- `build_q1_model`: 构造旅行项 + 双唯一性惩罚项
- `solve_tsp_qubo_candidates`: 多 seed 候选采样
- `decode_assignment`: 从 QUBO 二值解恢复合法排列
- `run_q1_baseline`: 自适应 lambda + profile ensemble + anchor 搜索总控

## 6. Q2 主线模型（最优可行版）

### 6.1 问题定义
- 15 客户单车路径，目标为 travel + 时间窗惩罚
- 时间窗口径：无等待服务（到达即服务）

### 6.2 主线模型
- 模型名称：`Enhanced direct-penalty QUBO + adaptive (lambda, tw_weight) + profile ensemble + anchor restarts`
- 关键设计：
  - 时间窗一元惩罚（位置级）直接入 QUBO
  - 相邻二元修正项（`x[i,t]x[j,t+1]`）
  - 二层自适应：先调 `lambda`，再调 `tw_weight`
  - 多 profile + 锚点重启提高搜索多样性

### 6.3 实验设置
- 客户数：15
- 关键参数：`seed_count=2`, `adaptive_rounds=5`, `adaptive_budget=6`, `tw_weight=1.0`
- 精确对照：DP（n<=15）
- 补充对照：Branch-Cut（统一口径）

### 6.4 结果报告
- 主线最优可行结果：
  - `travel=40, tw_penalty=175620, objective=175660, feasible=True`
  - 文件：`experiments/results/q2_20260418_133144.json`
- 相对旧基线（`objective=241172`）下降约 27.16%
- 质量锚点（对照报告）：
  - DP: `84121`
  - Branch-Cut: `97920`
  - QUBO 仍有 gap，但具备可扩展与可解释优势

### 6.5 代码实现原理
- `evaluate_time_window_penalty`: 统一评估口径
- `estimate_position_arrival_times` + `build_position_penalty_matrix`: 时间窗一元项估计
- `build_pairwise_penalty_tensor`: 相邻位置二元修正
- `build_q2_model_with_position_penalty`: 统一组装 QUBO
- `run_q2_baseline`: 二层自适应 + profile 集成 + 锚点重启总控

## 7. Q3 主线模型（当前最优 travel 版）

### 7.1 问题定义
- 50 客户单车路径，量子子问题规模受限（QUBO 子问题上限 15）

### 7.2 主线模型
- 模型名称：`decomposition (cluster) + QUBO subproblem + stitching + local 2-opt`
- 主线逻辑：
  - 分解：按簇切分客户
  - 簇内：QUBO 求解局部路径
  - 簇间：拼接并局部改进（2-opt / or-opt / tw-repair）

### 7.3 实验设置
- 客户数：50
- `cluster_size=10`
- `seed_count_per_cluster=3`
- `do_two_opt=True`

### 7.4 结果报告
- 当前最优 travel 结果：
  - `travel=62, tw_penalty=5425280, objective=5425342, feasible=True`
  - 文件：`experiments/results/q3_20260417_151446.json`
- 说明：另有 objective 更低的配置（travel=63），但本轮主线按 travel 优先选取

### 7.5 代码实现原理
- `partition_customers`: 分解策略（距离/时间窗/融合）
- `_run_single_strategy`: 簇内 QUBO 求解与日志
- `_postprocess_sequence`: 2-opt/or-opt/tw 修复
- `run_q3_baseline`: 单策略或融合策略汇总与可行性选择

## 8. Q4 主线模型（最新最优版）

### 8.1 问题定义
- 50 客户，多车、容量、时间窗、多目标（车辆数 + 路程 + 罚项）

### 8.2 主线模型
- 模型名称：`two-stage hybrid VRP: assignment + per-vehicle QUBO + repair + optional cross-vehicle refine`
- 两阶段框架：
  1. 分配层：客户分配到车辆（容量硬约束）
  2. 路由层：车内 QUBO 路径优化
- 增强点（本轮提分关键）：
  - 跨车重分配局部搜索（relocate/swap）
  - 选择策略新增 `travel_first`（可行优先后按 travel 选）

### 8.3 实验设置
- 客户数：50
- 关键参数（最优配置）：
  - `assignment_strategy=ffd`
  - `route_postprocess=or_opt`
  - `enable_tw_repair=True`
  - `enable_cross_vehicle_refine=True`
  - `cross_vehicle_max_iter=16`
  - `cross_vehicle_candidate_per_route=4`
  - `cross_vehicle_allow_swap=True`
  - `vehicle_scan_mode=travel_first`
  - `min_vehicle_count=5, max_vehicle_count=14`
  - `seed_count_per_vehicle=1`
  - `vehicle_weight=0`

### 8.4 结果报告
- 最新最优结果：
  - `travel=109, tw_penalty=51640, objective=51749, feasible=True`
  - 文件：`experiments/results/q4_20260419_120746.json`
- 对比提升：
  - 相对旧主线（`travel=131`）下降约 16.79%
  - 相对你给的参考目标 `117`：进一步下降 8
- 增强批次证据：`experiments/results/q4_boost_quick_batch_20260419.json`
  - Top-6 配置 travel 依次为：109, 110, 112, 114, 119, 124

### 8.5 代码实现原理
- `optimize_single_vehicle_route`: 每车 QUBO 求解 + 后处理
- `_cross_vehicle_refine_routes`: 跨车 relocate/swap 局部搜索（容量硬约束）
- `run_q4_baseline`: 车辆数扫描 + 策略选择 + 诊断输出
- `diagnostics.selected_refine_diagnostics`: 输出跨车迭代历史与收敛信息

## 10. 消融与关键决策结论

### 10.1 Q2
- 决策点：是否使用 B+C 迭代主线
- 结论：当前提交主线采用增强直入模（可行性与可复现实验链更完整），B+C 作为补充证据

### 10.2 Q3
- 决策点：分解粒度与后处理
- 结论：`cluster_size` 对结果敏感，局部修复（2-opt/or-opt）必须保留

### 10.3 Q4
- 决策点：分配策略、扫描规则、是否跨车协同
- 结论：跨车协同是本轮核心增益来源；`travel_first` 对“以 travel 为主目标”最有效
